# 18 · Sub-agents — 여러 AI 협업

## Sub-agent가 뭐예요?

> **Sub-agent (서브 에이전트) = 메인 Claude가 별도 작업을 맡기는 또 다른 Claude**

복잡한 일을 메인 한 명이 다 하지 않고, **전문가 Claude를 여러 명** 두는 구조.

비유: 대표 PM(메인 Claude) 밑에 **리서처, 코드 리뷰어, 디자인 검수자**가 따로 있는 식.

## 왜 쓰나요?

### 컨텍스트 분리
큰 코드베이스 검색 같은 일을 메인이 직접 하면 컨텍스트 폭발. **서브 에이전트가 검색만 하고 결과만 보고**하면 메인의 컨텍스트는 깨끗.

### 전문성 분리
- "보안 검토"용 에이전트는 보안 관점만
- "성능 검토"용 에이전트는 성능만
- 각자 본인 영역에서만 깊게 봄

### 병렬 진행
독립적인 일들은 동시에 여러 에이전트가 진행 → 시간 단축.

## 기본 사용

### 메인 Claude가 자동으로

대부분의 경우 메인 Claude가 알아서 판단:

```
> 이 모놀리스 코드베이스에서 인증 관련 함수들 다 찾아서 리스트업해줘.
```

→ 메인은 "이건 검색량이 많겠네" 라고 판단. 자동으로 **Explore 서브에이전트** 호출. 결과만 받아서 정리.

### 명시적으로 부르기

```
> 보안 리뷰 전문 에이전트 불러서 이 PR 검토 받아줘.
```

→ 메인이 `code-reviewer` 서브 에이전트에 일을 맡깁니다.

## 표준 서브 에이전트 종류

환경에 따라 다르지만 자주 보이는 것들:

| 에이전트 | 역할 |
|---|---|
| **Explore** | 코드베이스 검색 / 위치 찾기 |
| **Plan** | 구현 계획 수립 |
| **general-purpose** | 범용 (긴 리서치) |
| **code-reviewer** | 코드 리뷰 |
| **claude-code-guide** | Claude Code 자체 사용법 안내 |
| **statusline-setup** | 상태바 설정 |

## 직접 만들기

`.claude/agents/<이름>.md` 또는 `~/.claude/agents/<이름>.md`:

```markdown
---
name: korean-translator
description: 한국어 번역 전문 에이전트. 영문 문서를 자연스러운 한국어로.
tools: [Read, Write, Edit]
---

당신은 한국어 번역 전문가입니다.

- 직역보다 의역을 우선 (한국어로 자연스럽게)
- 기술 용어는 첫 등장에 영문 병기 (예: "컴포넌트(component)")
- 코드 블록 / 파일명 / 명령어는 번역하지 않음
- 어조는 정중한 평어체 ("~합니다")
```

호출:

```
> @agent korean-translator README.md를 번역해서 README_ko.md로 저장
```

## 메인이 서브에 일 맡기는 패턴 (개발자용)

서브에이전트 정의에서 사용 가능 도구를 제한할 수 있어요:

```yaml
tools: [Read, Grep, Glob]   # 읽기만 가능
```

```yaml
tools: '*'                  # 모든 도구 (위험)
```

이렇게 하면 **검색 전문 에이전트는 파일 수정 못 함** 같은 안전장치.

## 실전 워크플로우 예시

### 큰 리팩토링 프로젝트

```
[메인]
  └─ Explore: 변경 영향 범위 검색 (50개 파일 발견)
  └─ Plan: 안전한 변경 순서 설계
  └─ general-purpose 1: pages/ 부분 리팩토링
  └─ general-purpose 2: components/ 부분 리팩토링 (병렬)
  └─ code-reviewer: 결과 검토
  └─ [메인이 통합 + 사용자에 보고]
```

각 서브에이전트는 자기 컨텍스트에서만 일하고 결과만 메인에게 전달.

### 보안 + 성능 동시 검토

```
[메인]
  ├─ security-review (병렬)
  └─ performance-review (병렬)
       ↓
  [메인이 결과 통합]
```

## 비용 / 성능 영향

- 서브 에이전트도 **별도 토큰을 소모**합니다
- 결과만 받기 때문에 **메인 컨텍스트는 절약**
- 병렬 실행은 빠르지만 동시 토큰 사용량 ↑

> 💡 가벼운 작업까지 서브를 쓰지 마세요. **컨텍스트가 부담될 것 같은 작업** 위주로.

## 디버깅

### 서브 에이전트 호출 추적
```
/agents
```

설치된 에이전트 목록과 동작 상태.

### 응답이 이상할 때
- 에이전트 정의의 `description` 이 명확한지
- `tools` 가 너무 좁지 않은지
- 메인의 부탁이 모호하지 않은지

## 자주 묻는 질문

<details>
<summary><b>Q. 메인이 서브를 안 부르고 자기가 다 하려고 해요</b></summary>

부탁할 때 명시: "이 작업은 코드베이스 전반을 검색해야 하니 Explore 에이전트에게 맡겨줘."
</details>

<details>
<summary><b>Q. 서브 에이전트가 한 일을 메인이 어떻게 알아요?</b></summary>

서브가 일 끝나면 **요약 결과**를 메인에 반환. 메인은 그 요약만 보고 다음 단계 결정. 디테일은 모름 (그래서 서브가 결과를 잘 정리해야 함).
</details>

<details>
<summary><b>Q. 직접 만든 서브 에이전트가 작동 안 해요</b></summary>

1. 폴더 경로 (`.claude/agents/<이름>.md` or `~/.claude/agents/`)
2. frontmatter 필수 필드 (name, description)
3. Claude 재시작
4. `/agents` 로 등록 확인
</details>
