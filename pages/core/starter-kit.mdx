# 프로젝트 파일 설계 — 무엇을 만들어둬야 할까

> "CLAUDE.md는 알겠는데, 또 만들어두면 좋은 게 뭐가 있어?"

이 페이지는 **프로젝트 시작 시 한 번만 설계해두면 평생 편한 파일들**을 정리합니다. 백과사전처럼 펴보면서 본인 상황에 맞는 것만 골라가도 OK.

> 📚 **이 페이지 한 눈에**
> - Claude Code 관련 파일은 **4가지 카테고리**: 자동 읽힘 / 자동화 자산 / 사람용 문서 / 안전망
> - **단계별 시작 키트**: 🥚 최소(10분) → 🐣 권장(한 시간) → 🐦 풀(반나절)
> - 모든 핵심 파일에 **복붙 가능한 템플릿** 제공
>
> 🔗 함께 보면 좋은 페이지: [08 CLAUDE.md](/core/claude-md) · [✏️ 에디터 셋업](/extras/editor-setup) · [17 Skills](/advanced/skills) · [19 워크스페이스 설계](/advanced/workspace)

---

## 4가지 카테고리

```
1. Claude가 자동으로 읽는 파일       ← 이름·위치 정해져 있음
2. 자동화 자산 (Skills/Agents/MCP)   ← 만들수록 강해짐
3. 사람용 문서 (Claude는 @import)     ← 토큰 절약하면서 깊이
4. 안전망 (.gitignore / deny 등)     ← 한 번 사고 막으면 본전
```

이 순서대로 자세히 봅니다.

---

## 1️⃣ Claude가 자동으로 읽는 파일들

이름과 위치가 정해져 있어서 **거기 두기만 하면** Claude가 매 세션마다 알아서 인식. 가장 효과 큰 카테고리.

| 파일 | 위치 | 용도 |
|---|---|---|
| **`CLAUDE.md`** | 프로젝트 루트 | 이 프로젝트의 규칙·컨벤션·금지사항 |
| **`CLAUDE.local.md`** | 프로젝트 루트 | 본인만의 메모·실험. **반드시 `.gitignore` 등록** |
| **`~/.claude/CLAUDE.md`** | 홈 폴더 | 모든 프로젝트 공통 — 내 프로필·기본 선호 |
| **`.claude/settings.json`** | 프로젝트 또는 홈 | 권한 allow/deny, 후크, 환경 변수 |
| **`~/.claude/memory/MEMORY.md`** + 그 폴더의 `*.md` | 홈 | Claude가 **자동으로 쓰고 읽는** 장기 기억. 처음엔 손대지 말고 Claude가 채우게 두면 됨 |

### 계층 우선순위 (헷갈리면 여기 보세요)

여러 곳에 같은 내용이 있으면 **가까운 게 우선**:

```
프로젝트/CLAUDE.local.md   ← 가장 강함 (본인 전용)
프로젝트/CLAUDE.md         ← 그 다음 (팀 공유)
~/.claude/CLAUDE.md        ← 가장 약함 (전역 기본)
```

→ 자세히는 [08 CLAUDE.md — 어디에 만들어요?](/core/claude-md#어디에-만들어요)

## 2️⃣ 자동화 자산

만들면 **반복 작업이 한 줄 명령**으로 줄어드는 영역. 처음엔 한두 개로 시작, 필요할 때마다 추가.

| 파일 | 위치 | 용도 |
|---|---|---|
| **`.claude/skills/<이름>/SKILL.md`** | 프로젝트 또는 홈 | `/단축어` 호출되는 작업 레시피. 회의록 정리·안전한 커밋·블로그 초안 등 |
| **`.claude/agents/<이름>.md`** | 프로젝트 또는 홈 | 전문 서브에이전트. 보안 리뷰어·번역가·검색 전문 등 |
| **`.claude/mcp.json`** | 프로젝트 또는 홈 | GitHub/Slack/DB 같은 외부 서비스 연결 정의 |
| **`.claude/commands/`** (환경에 따라) | 프로젝트 | 간단한 커스텀 슬래시 명령 |

전역(`~/.claude/`)에 두면 모든 프로젝트에서, 프로젝트(`.claude/`)에 두면 그 프로젝트에서만 사용.

→ 자세히: [17 Skills](/advanced/skills) · [18 Sub-agents](/advanced/subagents) · [15 MCP](/advanced/mcp)

## 3️⃣ 사람용 문서 (Claude는 `@import`로 끌어옴)

Claude가 자동으로 읽지는 않지만 **CLAUDE.md에서 `@docs/conventions.md` 처럼 적어두면** 필요할 때만 펴봅니다. → 토큰 절약 + 분리.

| 파일 | 무엇 | 누가 자주 봄 |
|---|---|---|
| **`README.md`** | 프로젝트 인트로 — 무엇이고, 왜 만들었고, 어떻게 시작 | 사람·Claude 둘 다 |
| **`docs/glossary.md`** | 이 도메인의 용어 사전 ("리드 = 영업 단서") | 사람·Claude |
| **`docs/conventions.md`** | 자세한 코딩/네이밍/커밋 규칙 | Claude (CLAUDE.md에서 가져옴) |
| **`docs/architecture.md`** | 시스템 구조 / 다이어그램 (간단 ASCII도 OK) | 사람·Claude |
| **`docs/decisions/<날짜>-<주제>.md`** | ADR (Architecture Decision Record) — **왜 이렇게 짰는지** 결정 로그 | 미래의 본인 |
| **`PLAYBOOK.md`** (또는 `RUNBOOK.md`) | "배포 프로세스", "DB 마이그레이션" 같은 자주 하는 작업 매뉴얼 | 본인·동료·Claude |
| **`docs/feature-flags.md`** | 활성/비활성 플래그, 어디서 조작하는지 | 운영 시 |
| **`CHANGELOG.md`** | 버전별 변경사항 (배포하는 프로젝트라면) | 사용자·동료 |

> 💡 **ADR이 뭐예요?** "이 기술을 왜 골랐는지" 같은 결정을 짧게 기록하는 문서. 6개월 뒤 본인도 까먹는 결정의 배경이 살아있어요. **3~10줄 짜리** 짧은 메모로 시작.

---

## 4️⃣ 안전망 — 작지만 사고 한 번 막으면 본전

| 파일 | 무엇 |
|---|---|
| **`.gitignore`** | `.env`, `node_modules`, 빌드 산출물, **`CLAUDE.local.md`**, **`.claude/secrets.*`** 등 git에 안 올림 |
| **`.claude/settings.json`** 의 `deny` 리스트 | `rm -rf *`, `git push --force*`, `Edit(.env)` 등 위험 작업 차단 |
| **`.env.example`** | 진짜 `.env` 대신 키 이름만 있는 템플릿. 새 팀원/Claude가 "어떤 키 필요한지" 알 수 있게 |
| **Hooks** (PreToolUse 차단) | 정말 위험한 명령은 settings deny + Hooks 이중 방어 |

→ Hooks 자세히: [16 Hooks](/advanced/hooks)

## 단계별 시작 키트

처음부터 다 만들지 마세요. 단계로 깔면 부담 없습니다.

### 🥚 최소 (10분, 모든 프로젝트)

이 세 개만 있어도 기본은 갖춥니다.

```
프로젝트/
├── CLAUDE.md
├── .gitignore
└── .claude/
    └── settings.json
```

### 🐣 권장 (한 시간, 진지한 프로젝트)

위 + 다음:

```
프로젝트/
├── CLAUDE.md
├── CLAUDE.local.md          ← .gitignore에 등록
├── README.md
├── .gitignore
├── .env.example             ← 환경변수 사용한다면
└── .claude/
    ├── settings.json
    └── skills/
        └── git-clean/
            └── SKILL.md     ← 첫 스킬 하나
```

그리고 전역에:
```
~/.claude/
└── CLAUDE.md                ← 내 프로필 (모든 프로젝트 공통)
```

### 🐦 풀 (반나절, 팀 또는 큰 프로젝트)

위 + 다음:

```
프로젝트/
├── CLAUDE.md
├── CLAUDE.local.md
├── README.md
├── PLAYBOOK.md
├── .gitignore
├── .env.example
├── docs/
│   ├── glossary.md
│   ├── conventions.md
│   ├── architecture.md
│   └── decisions/
│       └── 2026-05-06-tech-stack.md
└── .claude/
    ├── settings.json
    ├── mcp.json             ← GitHub MCP 등
    ├── skills/
    │   ├── git-clean/SKILL.md
    │   ├── code-review/SKILL.md
    │   └── deploy/SKILL.md
    └── agents/
        ├── code-reviewer.md
        └── security-review.md
```

## 복붙 템플릿

위 파일들 중 핵심을 **그대로 복사해서 시작**할 수 있게 모아뒀어요.

### `CLAUDE.md` (프로젝트 루트)

```markdown
# 프로젝트 개요
[한 줄로 무엇을 하는 곳인지]

# 기술 스택
- 언어/프레임워크: 
- 배포: 
- DB: 

# 코드 스타일
- 함수명: 영어 camelCase
- 주석: 한국어
- import 순서: 외부 → 내부 → 상대경로

# 자주 쓰는 명령
- 개발 서버: `npm run dev`
- 빌드: `npm run build`
- 테스트: `npm test`

# 절대 하지 마세요
- main 브랜치에 직접 push 금지
- .env, *.key 파일 수정/커밋 금지
- 큰 변경 전엔 git commit으로 백업

# 자주 묻는 폴더 구조
- src/ : 소스 코드
- docs/ : 문서
- .claude/ : Claude Code 설정

# 자세한 컨벤션
@docs/conventions.md 참조
```

### `CLAUDE.local.md` (본인 전용, gitignore!)

```markdown
# 내 작업 메모
- 지금 작업 중인 브랜치: feature/dark-mode
- 이번 주 끝낼 일: 다크모드 + 모바일 메뉴

# 내 환경
- DB는 docker로 로컬에서 (localhost:5432)
- API 키는 ~/.config/myproject/local.env 에

# 실험 중인 것 (정식 채택 전)
- bun 으로 빌드 속도 비교 중
```

### `~/.claude/CLAUDE.md` (전역 — 모든 프로젝트 공통)

```markdown
# 사용자 정보
- 이름: [본인 이름]
- 역할: [예: 1인 개발자, 마케터, 데이터 분석가]
- 주 사용 OS: macOS / Windows / Linux

# 일반 선호
- 답변 언어: 한국어
- 코드 주석: 한국어 / 함수·변수명: 영어
- 큰 변경 전엔 항상 git commit으로 백업
- 위험한 작업(rm, force push)은 항상 확인 받기

# 작업 스타일
- 작은 단위로 자주 검증
- 큰 작업은 계획만 먼저 보여주고 OK 받은 뒤 진행
- 모르는 부분은 추측하지 말고 물어보기
```

### `.claude/settings.json` (프로젝트)

```json
{
  "permissions": {
    "allow": [
      "Read(**)",
      "Bash(git status)",
      "Bash(git log*)",
      "Bash(git diff*)",
      "Bash(npm run *)",
      "Bash(ls*)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force*)",
      "Bash(git reset --hard*)",
      "Edit(.env)",
      "Edit(**/*.key)",
      "Edit(**/secrets.*)"
    ],
    "defaultMode": "default"
  }
}
```

### `docs/conventions.md` (자세한 컨벤션)

```markdown
# 코딩 컨벤션

## 네이밍

### 변수/함수
- `camelCase`: 일반 변수, 함수
- `PascalCase`: 컴포넌트, 클래스
- `SCREAMING_SNAKE_CASE`: 상수

### 파일명
- 컴포넌트: `PascalCase.tsx`
- 유틸리티: `kebab-case.ts`
- 테스트: `*.test.ts` (옆에 둠)

## 함수 작성

### 단일 책임
한 함수는 한 가지만 해야 합니다.

\`\`\`typescript
// ❌ 두 가지 함
function fetchAndSaveUser(id) {...}

// ✅ 분리
function fetchUser(id) {...}
function saveUser(user) {...}
\`\`\`

### 에러 처리
- 외부 호출은 try/catch
- 비즈니스 로직 실패는 명시적 결과 객체

## Git 컨벤션

### 커밋 메시지 (Conventional Commits)
- `feat:` 새 기능
- `fix:` 버그 수정
- `refactor:` 리팩토링
- `docs:` 문서
- `test:` 테스트
- `chore:` 잡일

### 브랜치
- `main` — 배포
- `develop` — 개발 통합
- `feature/<설명>` — 기능
- `fix/<설명>` — 버그
```

### `docs/decisions/2026-05-06-tech-stack.md` (ADR 예시)

```markdown
# ADR-001: 기술 스택 선정

날짜: 2026-05-06
상태: 채택

## 배경
한국 카페 추천 사이트. SEO 중요, 빠른 첫 출시 필요.

## 결정
- 프레임워크: Next.js 14 (App Router)
- 스타일: Tailwind CSS
- DB: PostgreSQL (Supabase)
- 배포: Railway

## 이유
- Next.js — SSR 기본 + 한국 사용자 빠른 로딩
- Tailwind — 작은 팀에서 디자인 속도
- Supabase — DB + Auth + Storage 한 번에
- Railway — git push만으로 배포

## 대안과 기각 이유
- Remix: 사용자 풀이 작아 채용 어려움
- Firebase: 한국 region 없음

## 영향
- 모든 컴포넌트는 RSC 우선, 'use client'는 명시적으로
- 결제는 별도 SaaS (Toss) 분리
```

처음엔 짧아도 됩니다. **6개월 뒤 본인이 "왜 이걸 썼지?" 안 헤매게.**

```markdown
## DB 마이그레이션
```

```markdown
---
name: git-clean
description: 변경사항 검토 + 안전한 커밋 + 푸시

# Git 안전 커밋

이 스킬은 다음을 차례로 합니다:

1. `git status` 로 변경 파일 확인
2. 변경된 파일에 `.env`, `*.key` 같은 민감 파일이 있는지 검사. 있으면 중단하고 사용자에게 보고
3. `git diff` 로 사용자에게 변경 미리보기
4. 변경사항 기반으로 적절한 커밋 메시지 자동 생성
   - 형식: Conventional Commits (`feat:`, `fix:` 등)
5. 사용자 승인을 받은 후에만 commit + push 진행
6. 결과 (커밋 해시, 푸시된 브랜치) 보고
```

```
/git-clean
```

## 안티패턴 (자주 빠지는 함정)

### ❌ CLAUDE.md에 모든 걸 한꺼번에

500줄 짜리 CLAUDE.md = 매 세션마다 500줄어치 토큰 낭비. **핵심만** CLAUDE.md, 자세한 건 `docs/` 분리 → `@import`.

### ❌ `CLAUDE.local.md`를 gitignore에 안 넣기

본인 비밀번호·실험 메모가 팀 레포에 올라감. 처음부터 `.gitignore`에 등록!

### ❌ `.env`를 `.env.example` 없이 쓰기

새 팀원/Claude가 어떤 키 필요한지 모름. 진짜 값 없이 키 이름만 있는 `.env.example` 항상 동반.

### ❌ Skills를 너무 일찍 많이

만든 스킬을 안 쓰면 의미 없음. **두 번 같은 일 했을 때 → CLAUDE.md 규칙. 세 번째 → Skills.**

### ❌ ADR을 길게 쓰려 함

3줄짜리 ADR도 가치 있음. **결정 + 이유 + 대안 기각 사유**만 적어도 충분.

### ❌ 전역(`~/.claude/`)에 프로젝트별 규칙

전역에 "Next.js 컨벤션" 적으면 다른 프로젝트(Python)에서도 적용됨 → 혼란. **전역은 본인 프로필만**, 프로젝트 규칙은 `.claude/`에.
